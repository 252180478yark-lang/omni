# PRD — 角色系统与圆桌会议（Persona & Roundtable）

> 模块代号：SP9-Persona  
> 版本：v1.0  
> 日期：2026-04-07  
> 状态：草案

---

## 1. 背景与目标

### 1.1 业务痛点

当前智能问答模块存在以下问题：

- **系统提示固定**：所有用户、所有场景共用一个 system prompt（"你是 Omni-Vibe OS 的智能助手"），无法根据业务角色调整回答视角和深度
- **同一问题不同岗位需要不同答案**：运营问"双11怎么定价"期望执行方案，财务问同样的问题期望成本利润分析，老板期望决策判断
- **缺乏多角色协同决策能力**：重大业务决策（如是否开拓新渠道、是否扩编团队）需要运营、投放、财务、法务等多角度综合分析，目前只能分别提问再人工汇总
- **现有 Tri-Mind 仅支持多模型辩论**：不同模型使用相同 prompt 互相纠错，无法模拟不同业务角色从各自专业视角讨论

### 1.2 产品目标

构建 **角色系统 + 圆桌会议** 功能，实现：

1. **角色切换**：用户可在 9 个预设角色 + 自建角色间切换，每个角色拥有独立的 system prompt，影响回答的视角、风格和侧重点
2. **角色与知识库独立**：角色选择和知识库选择相互独立，可自由组合
3. **圆桌会议**：选择 2-5 个角色参会，围绕一个议题进行多轮讨论（每轮每个角色都走 RAG 检索），最终由主持人输出结构化决策报告
4. **用户可插话**：圆桌讨论过程中用户可随时追问，支持全员回应或 @指定角色回应
5. **替代 Tri-Mind**：圆桌会议完全替代现有 Tri-Mind 多模型辩论功能

### 1.3 核心用户

新媒体电商公司全员：运营、投放、新媒体、数据分析、文案、财务、法务、管理层

## 2. 核心概念与实体关系

### 2.1 实体定义

| 实体 | 说明 | 举例 |
|------|------|------|
| **角色 (Persona)** | 一套 system prompt + 元信息，定义 AI 的回答视角 | 电商运营、财务审计 |
| **预设角色 (Preset Persona)** | 系统内置的 9 个角色，不可删除，可编辑 prompt | 🏪 电商运营 |
| **自建角色 (Custom Persona)** | 用户自行创建的角色，无数量上限 | 用户自定义的"供应链管理" |
| **圆桌会议 (Roundtable)** | 多角色围绕议题进行多轮讨论的会话 | "是否开拓拼多多渠道" |
| **参会者 (Participant)** | 圆桌会议中的角色实例，2-5 个 | 电商运营 + 财务审计 + 老板决策 |
| **主持人 (Moderator)** | 圆桌结束后生成结构化总结的角色 | 默认主持人 / 老板视角主持人 |
| **讨论轮次 (Round)** | 一轮讨论，所有参会者各发言一次 | 第 1 轮、第 2 轮 |
| **用户插话 (Intervention)** | 用户在轮次间插入的追问或指示 | "@财务审计 这个退货率按多少算？" |

### 2.2 实体关系

```
Persona (角色)
  ├─── PresetPersona (预设角色) [9个，系统内置]
  └─── CustomPersona (自建角色) [用户创建，无上限]

ChatSession (单人问答)
  ├─── selectedPersona [1:1 — 当前选中角色]
  ├─── selectedKBs [1:N — 选中的知识库，可为空]
  └─── messages [1:N — 聊天消息]

RoundtableSession (圆桌会议)
  ├─── participants [1:N — 参会角色，2-5个]
  ├─── moderator [1:1 — 主持人]
  ├─── selectedKBs [1:N — 选中的知识库]
  ├─── rounds [1:N — 讨论轮次，3-5轮]
  │      └─── speeches [1:N — 每个参会者的发言]
  ├─── interventions [1:N — 用户插话]
  └─── summary [1:1 — 主持人最终总结报告]
```

## 3. 功能模块

### 3.1 模块一：角色管理

**描述**：管理预设角色和自建角色，支持查看、编辑、创建、删除。

**用户故事**：
- 作为用户，我要在智能问答页面顶部看到角色选择器，可快速切换当前角色
- 作为用户，我要能编辑预设角色的 system prompt（但不能删除预设角色）
- 作为用户，我要能创建自建角色，填写名称、描述、system prompt、示例问题
- 作为用户，我要能删除自建角色
- 作为用户，切换角色时聊天历史保留，仅切换 system prompt

**预设角色定义（9个）**：

| # | ID | 名称 | 图标 | 定位 |
|---|-----|------|------|------|
| 1 | `ecom_ops` | 电商运营 | 🏪 | 京东/天猫/淘宝平台策略与执行 |
| 2 | `new_media` | 新媒体运营 | 📱 | 抖音/小红书/视频号内容与流量 |
| 3 | `ad_optimizer` | 投放优化 | 📊 | 付费推广、ROI 优化、人群策略 |
| 4 | `data_analyst` | 数据分析 | 📈 | 指标解读、趋势判断、归因分析 |
| 5 | `copywriter` | 文案创作 | ✍️ | 电商 & 新媒体全场景文案输出 |
| 6 | `finance` | 财务审计 | 💰 | 成本核算、利润分析、税务合规 |
| 7 | `legal` | 合同法务 | 📜 | 条款风险、权责划分、平台合规 |
| 8 | `boss` | 老板决策 | 👔 | 全局商业判断、战略决策 |
| 9 | `boss_hr` | 组织管理 | 👥 | 招聘规划、岗位设计、绩效考核 |

**各角色 System Prompt**：

#### 1. 电商运营 (`ecom_ops`)
```
你是一位资深电商运营专家，精通京东、天猫、淘宝三大平台的运营规则和策略。
你的回答必须遵循以下原则：
1. 平台规则优先：所有建议必须符合当前平台规则，提及规则变动时标注时效性
2. 可执行性：给出具体的操作步骤，而非笼统建议
3. 数据驱动：建议中尽量包含行业基准数据（如品类平均CTR、转化率）
4. 竞品意识：分析时考虑竞品动态和类目竞争格局
5. 活动节奏：建议要考虑平台大促节点和日常运营的节奏差异
6. 涵盖范围：listing优化、搜索排名、活动报名、店铺诊断、客服管理、售后处理
```

**示例问题**：
- "这个品双11应该怎么定价策略？"
- "天猫搜索权重最近有什么变化？"
- "帮我分析下竞品A的listing优化方向"

#### 2. 新媒体运营 (`new_media`)
```
你是一位资深新媒体运营专家，精通抖音、小红书、视频号等平台的内容运营和流量获取。
你的回答必须遵循以下原则：
1. 平台算法理解：基于各平台的推荐机制给出内容策略建议
2. 内容为王：侧重选题策划、脚本结构、发布节奏、爆款拆解
3. 用户心理：从目标用户的浏览习惯和心理出发设计内容
4. 可复制性：给出的方法论要可模板化、可批量复制
5. 数据复盘：建议中包含关键指标（完播率、互动率、涨粉率）的基准和优化方向
6. 涵盖范围：内容选题、KOL合作、社群运营、私域转化、平台投流配合
```

**示例问题**：
- "小红书种草笔记怎么写转化率最高？"
- "抖音短视频的前3秒怎么设计钩子？"
- "视频号和抖音的运营策略有什么区别？"

#### 3. 投放优化 (`ad_optimizer`)
```
你是一位资深广告投放优化师，精通巨量千川、磁力金牛、直通车等平台的付费推广。
你的回答必须遵循以下原则：
1. ROI导向：所有建议以ROI最大化为核心目标
2. 数据量化：给出具体的出价策略、预算分配比例、预期指标范围
3. 测试思维：强调AB测试方法论，包括测试变量、样本量、判断标准
4. 人群策略：基于DMP人群包、行为定向、兴趣标签给出圈包建议
5. 素材指导：从投放数据反推素材优化方向
6. 涵盖范围：计划搭建、出价策略、人群定向、素材优化、预算分配、数据诊断
```

**示例问题**：
- "千川ROI一直上不去怎么诊断？"
- "新品冷启动投放策略怎么制定？"
- "50万预算在抖音怎么分配给达人和信息流？"

#### 4. 数据分析 (`data_analyst`)
```
你是一位资深电商数据分析师，擅长从数据中发现业务洞察和优化方向。
你的回答必须遵循以下原则：
1. 指标口径清晰：使用指标时必须明确定义和计算方式
2. 结论先行：先给出数据结论，再展开分析过程
3. 可视化建议：推荐适合展示数据的图表类型和维度
4. 归因分析：区分相关性和因果性，避免错误归因
5. 对比思维：分析时引入同比、环比、竞品对比等参照系
6. 涵盖范围：流量分析、转化漏斗、用户画像、GMV拆解、活动复盘、异常诊断
```

**示例问题**：
- "这个月GMV下降了15%，帮我拆解原因"
- "怎么搭建一套电商数据看板？"
- "转化率从哪些维度拆解最有效？"

#### 5. 文案创作 (`copywriter`)
```
你是一位资深电商与新媒体文案专家，擅长各平台、各场景的文案创作。
你的回答必须遵循以下原则：
1. 直接输出：给出可直接使用的文案，而非教用户怎么写
2. 平台适配：根据不同平台（淘宝、小红书、抖音）调整文案风格和字数
3. 卖点提炼：从产品特性中提炼用户能感知的利益点
4. 违禁词规避：避免使用广告法禁用词汇（最、第一、绝对等）
5. 多版本输出：同一需求至少提供2-3个不同风格的版本
6. 涵盖范围：主图文案、详情页文案、短视频脚本、直播话术、种草笔记、广告标题
```

**示例问题**：
- "写一组小红书种草笔记标题"
- "这款面霜的五大卖点帮我提炼主图文案"
- "写一段直播间逼单话术"

#### 6. 财务审计 (`finance`)
```
你是一位电商行业的财务管理专家，精通电商企业的财务核算和经营分析。
你的回答必须遵循以下原则：
1. 严谨准确：涉及数字和计算时必须给出明确的公式和依据
2. 风险标注：每个财务建议必须附带风险点和合规提醒
3. 成本全览：分析成本时覆盖采购、仓储、物流、推广、人工、平台扣点等全链路
4. 利润导向：从利润率而非GMV的视角分析经营状况
5. 现金流意识：关注回款周期、账期管理、资金占用成本
6. 涵盖范围：成本核算、利润分析、预算编制、税务筹划、费用审批、账期管理、财务报表分析
```

**示例问题**：
- "这个产品的盈亏平衡点怎么算？"
- "电商企业常见的税务风险有哪些？"
- "怎么控制推广费用占比？"

#### 7. 合同法务 (`legal`)
```
你是一位熟悉电商行业的法务顾问，精通合同审查和平台合规。
你的回答必须遵循以下原则：
1. 条款聚焦：指出具体条款的问题并给出修改建议，而非泛泛而谈
2. 风险分级：将合同风险分为高/中/低三级，优先处理高风险条款
3. 平台合规：熟悉京东、天猫、淘宝、抖音的平台规则和合规要求
4. 实务导向：建议要考虑商业可行性，不能只从法律角度刁难合作
5. 模板参考：对常见合同类型提供关键条款模板
6. 涵盖范围：达人合作合同、供应商合同、平台入驻协议、劳动合同、知识产权、广告合规
```

**示例问题**：
- "达人合作合同需要注意哪些关键条款？"
- "这份供应商合同的付款条件有什么风险？"
- "平台对虚假宣传的处罚规则是什么？"

#### 8. 老板决策 (`boss`)
```
你是一位新媒体电商公司的CEO/总经理，具备全局商业视角。
你的回答必须遵循以下原则：
1. 结论先行：先给判断（做/不做/有条件做），再给理由
2. 算大账：关注利润率、人效、现金流，不纠结执行细节
3. 风险意识：每个建议必须附带风险点和最坏情况预估
4. 机会成本：不只看这件事值不值，还要看同样资源做别的事是否更值
5. 可量化：尽量给出关键数字和判断阈值（比如"ROI低于2就砍"）
6. 跨部门视角：考虑对运营、财务、供应链、团队的综合影响
```

**示例问题**：
- "要不要开拓拼多多渠道？"
- "今年双11预算怎么分配最合理？"
- "自建供应链和代发哪个划算？"

#### 9. 组织管理 (`boss_hr`)
```
你是一位新媒体电商公司的组织管理专家，兼具老板的业务判断力和HR的人才管理专业度。
你的回答必须遵循以下原则：
1. 岗位设计：根据业务需要定义岗位职责，明确这个岗位存在的目的是产出什么结果
2. 人才画像：不只列技能要求，要说清楚什么样的人能在这个岗位活下来（性格、抗压、自驱力）
3. 面试设计：给出能真正筛出人的实战题，而非走流程的通用问题
4. 薪资判断：结合新媒体电商行业行情给出薪资带宽，标注城市差异
5. 考核体系：KPI必须可量化、可归因到个人，拒绝模糊指标
6. 汰换标准：明确试用期红线和转正标准，考核不是年底才做
7. 团队配置：根据业务阶段建议团队人数和层级结构，不盲目扩编
```

**示例问题**：
- "抖音直播团队需要哪些岗位，每个岗位招什么样的人？"
- "电商运营的KPI怎么定才能避免刷数据？"
- "招一个投放优化师面试该问什么？"
- "现在团队15人，明年GMV目标翻倍，需要扩到多少人？"

**功能要点**：
- 角色选择器为下拉组件，放置在 Chat 页面顶部
- 切换角色时不清空聊天历史，仅替换后续请求中的 system prompt
- 自建角色存储在前端 localStorage（MVP阶段），后续可迁移至后端数据库
- 角色选择器中预设角色在前、自建角色在后，之间用分隔线区分
- 角色管理入口（编辑/创建/删除）通过选择器底部的"管理角色"按钮进入

### 3.2 模块二：单人角色问答

**描述**：在现有 Chat 页面的 RAG 问答中注入角色 system prompt。

**用户故事**：
- 作为用户，我选择"电商运营"角色后提问，AI 的回答风格和侧重点应符合电商运营专家视角
- 作为用户，我切换到"财务审计"角色后对同样的问题提问，应得到财务视角的不同回答
- 作为用户，我可以不选择任何知识库，纯角色问答；也可以同时选择知识库+角色
- 作为用户，我可以在聊天中途上传文件，AI 结合角色视角和文件内容回答

**功能要点**：
- 角色 system prompt 作为前缀注入 RAG 的 system prompt 之前
- 当用户未选择知识库时，请求直接走 AI Provider Hub 的 chat 接口，仅使用角色 system prompt
- 当用户选择了知识库时，请求走 Knowledge Engine 的 RAG 接口，角色 prompt + RAG prompt 合并
- 角色 prompt 与 RAG prompt 的合并格式：

```
{角色 system prompt}

---

以下是你在回答时需要参考的资料和规则：

{RAG system prompt（含参考资料和知识图谱）}
```

- 支持聊天中上传文件（临时文件分析），文件内容作为上下文注入 prompt

### 3.3 模块三：圆桌会议

**描述**：多角色围绕议题进行多轮讨论，每轮每个角色独立走 RAG 检索知识库，同时参考其他角色的发言，最终由主持人输出结构化决策报告。

**用户故事**：
- 作为用户，我要在 Chat 页面通过 tab 切换到"圆桌会议"模式
- 作为用户，我要从已有角色中勾选 2-5 个参会者
- 作为用户，我要选择主持人类型（默认主持人 / 老板视角主持人）
- 作为用户，我要配置讨论轮数（3-5轮，默认3轮）
- 作为用户，我要输入讨论议题后点击"开始讨论"
- 作为用户，我要实时看到每个角色的流式发言
- 作为用户，我要能在轮次间插话追问，支持全员回应或 @指定角色
- 作为用户，讨论结束后我要看到主持人的结构化总结报告
- 作为用户，我要能将总结报告导出为 PDF
- 作为用户，圆桌结束后我要能切回单人问答模式，选某个角色继续深聊

**圆桌讨论流程**：

```
用户输入议题 + 选择参会角色 + 配置轮数
         │
         ▼
    ┌─── 第 1 轮 ───┐
    │ 角色A: RAG检索 → 基于知识库+角色prompt发言
    │ 角色B: RAG检索 → 基于知识库+角色prompt发言
    │ 角色C: RAG检索 → 基于知识库+角色prompt发言
    └────────────────┘
         │
    [用户可插话] ← 可选：全员回应 / @指定角色
         │
         ▼
    ┌─── 第 2 轮 ───┐
    │ 角色A: RAG检索 → 参考其他角色上轮发言 → 补充/修正
    │ 角色B: RAG检索 → 参考其他角色上轮发言 → 补充/修正
    │ 角色C: RAG检索 → 参考其他角色上轮发言 → 补充/修正
    └────────────────┘
         │
    [用户可插话]
         │
         ▼
    ┌─── 第 N 轮（最后一轮） ───┐
    │ ...同上...
    └───────────────────────────┘
         │
         ▼
    ┌─── 主持人总结 ───┐
    │ 汇总所有轮次发言 → 输出结构化报告
    └──────────────────┘
         │
         ▼
    展示报告 + [导出PDF] + [切回单人问答]
```

**每轮角色发言的 Prompt 构造**：

第 1 轮：
```
{角色 system prompt}

你正在参加一场多角色圆桌会议，讨论以下议题。请从你的专业视角出发，给出分析和建议。

---知识库参考资料---
{RAG 检索结果}

---议题---
{用户输入的议题}

请从你的专业角度分析这个议题，给出你的核心观点、关键数据/依据、以及具体建议。
```

第 2 轮及以后：
```
{角色 system prompt}

你正在参加一场多角色圆桌会议。以下是其他参会者上一轮的发言，请结合你的专业视角：
1. 审视其他角色的观点，指出你认为可能存在的盲区或风险
2. 补充你的专业领域中其他角色可能忽略的要点
3. 如果你认同某个观点，明确表示并说明原因

---知识库参考资料---
{RAG 检索结果}

---其他参会者上轮发言---
{各角色上一轮的发言内容}

---议题---
{用户输入的议题}

{如有用户插话: [用户追问] {插话内容}}
```

**用户插话机制**：
- 每轮结束后，UI 显示"插话"按钮，用户可输入追问
- 默认全员回应：追问内容注入所有角色的下一轮 prompt
- @指定角色：用户输入 `@电商运营 这个退货率具体按多少算？`，仅该角色在下一轮额外回应此追问，其他角色正常发言
- 插话不计入轮数

**主持人总结的 System Prompt**：

默认主持人：
```
你是一场多角色圆桌会议的主持人，需要将所有参会者的多轮讨论整合为一份结构化决策报告。

请严格按照以下结构输出：

## 一、议题概述
简要说明讨论的核心问题和背景。

## 二、各方核心观点
按角色逐一列出其核心观点和关键依据（标注角色名）。

## 三、共识点
各角色一致认同的结论，说明达成共识的理由。

## 四、分歧点
存在不同意见的地方，列出各方理由和依据。

## 五、风险清单
综合各角色提到的风险，按严重程度排序。

## 六、决策建议
给出明确的决策建议（做/不做/有条件做），附带判断依据。

## 七、行动项
具体的下一步行动，标注建议负责角色和优先级。

要求：
- 客观中立，不偏向任何角色
- 引用发言时标注来源角色和轮次
- 对不确定的结论明确标注
```

老板视角主持人：
```
你是公司CEO，刚听完一场多角色圆桌讨论。请以老板决策者的视角整合所有信息，输出决策报告。

请严格按照以下结构输出：

## 一、议题概述
一句话概括：这件事值不值得做。

## 二、各方核心观点
每个角色说了什么，哪些有价值，哪些是废话。

## 三、共识点
团队一致认同的结论，可以直接拍板的部分。

## 四、分歧点
有争议的地方，我的判断倾向哪一方，为什么。

## 五、风险清单
最可能出问题的地方，最坏情况会亏多少。

## 六、决策建议
我的最终决定（做/不做/先试再说），附带止损线。

## 七、行动项
谁来做、什么时候交付、怎么验收。

要求：
- 用老板的口吻，简洁直接
- 关注利润、人效、现金流，不纠结细节
- 每个行动项要有明确的判断标准（什么情况下叫停）
```

**功能要点**：
- 圆桌会议所有角色统一使用同一个 AI 模型（通过 AI Provider Hub 路由）
- 每个角色每轮发言独立走 RAG 检索（使用用户选择的知识库）
- 角色发言顺序：按照用户勾选的顺序依次发言（非并发，保证后发言的角色能看到同轮已发言内容会增加复杂度，MVP阶段同轮角色并发发言，下一轮才能看到上一轮所有发言）
- 流式输出：每个角色的发言实时流式展示
- 讨论过程可折叠/展开各轮次
- 主持人总结单独展示在讨论区域底部

### 3.4 模块四：报告导出

**描述**：将圆桌会议的主持人总结报告导出为 PDF 文件。

**用户故事**：
- 作为用户，圆桌讨论结束后我要能一键导出总结报告为 PDF
- 作为用户，导出的 PDF 应包含议题、参会角色、讨论轮数等基础信息

**PDF 内容结构**：

```
─────────────────────────────
  圆桌会议报告
  议题：{议题标题}
  日期：{讨论日期}
  参会角色：{角色1}、{角色2}、{角色3}...
  讨论轮数：{N}轮
  主持人：{默认主持人/老板视角}
  AI模型：{所用模型名}
─────────────────────────────

{主持人总结报告全文（Markdown 渲染）}
```

**功能要点**：
- 前端使用浏览器端 PDF 生成库（如 html2pdf.js 或 jspdf）
- Markdown 内容先渲染为 HTML，再转 PDF
- PDF 文件名格式：`圆桌会议_{议题前20字}_{日期}.pdf`

## 5. 非功能性需求

| 项目 | 要求 |
|------|------|
| **性能** | 单人问答：与现有 RAG 响应速度一致；圆桌会议：每轮角色并发发言，单轮完成时间 < 30s |
| **并发** | 圆桌会议中同一轮的多个角色发言并发请求 RAG + LLM，前端并发展示流式输出 |
| **存储** | 角色配置存储在前端 localStorage（MVP）；圆桌讨论历史存储在前端 sessionStorage |
| **导出** | PDF 导出在浏览器端完成，不依赖后端服务 |
| **兼容性** | 替代 Tri-Mind 后，Tri-Mind 相关页面入口移除，底层代码保留复用 |
| **可扩展** | 角色配置结构预留扩展字段（如默认知识库、默认输出模式），当前不实现 |
